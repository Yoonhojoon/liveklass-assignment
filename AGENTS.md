# Liveklass Demo Agent Guide

이 저장소는 **과제 C — 알림 발송 시스템** 구현용 프로젝트입니다.
작업 전 이 파일을 목차처럼 보고, 필요한 요구사항 문서와 하네스를 함께 확인하세요.

## 문서 위치

- 문서 디렉터리: `docs/requirements`
- 문서 인덱스: `docs/requirements/assignments.md`
- 작업 하네스: `harness/`
- 프로젝트 범위 Codex 에이전트/스킬: `.codex/agents/`, `.codex/skills/`

## 작업 전 로딩 규칙

과제 C 구현, 검증, README 정리, 제출 준비 작업을 시작하기 전에 다음 하네스 문서를 먼저 읽고 현재 작업 기준으로 적용하세요.

1. `harness/workflow.md`
2. `harness/controls.md`
3. `harness/assignment-c-checklist.md`
4. `harness/package-structure.md`

`.codex/agents/*.toml`은 기본으로 전부 읽는 문서가 아니라 역할별 위임 지침입니다. 해당 역할의 Codex native subagent를 사용할 때 적용하세요.
`.codex/skills/*/SKILL.md`는 기본으로 전부 읽는 문서가 아니라 반복 작업용 스킬입니다. 관련 작업을 수행하거나 스킬을 명시 호출했을 때 해당 `SKILL.md`를 읽고 따르세요.

## 구현 기준

- 알림 발송은 API 요청 스레드와 분리합니다.
- 알림 실패가 원래 비즈니스 트랜잭션을 실패시키면 안 됩니다.
- 실패 예외는 무시하지 말고 상태, 실패 사유, 재시도 정보로 저장합니다.
- 동일 이벤트 알림은 중복 발송하지 않습니다.
- 중복 방지는 애플리케이션 로직과 DB 유니크 제약을 함께 사용합니다.
- 서버 재시작 후 `REQUESTED`, 재시도 시각이 지난 `RETRY_WAITING`, 오래 머문 `PROCESSING` 요청을 복구 대상으로 봅니다.
- 다중 인스턴스 환경에서는 한 요청을 한 워커만 점유하도록 원자적 상태 전이를 사용합니다.
- 실제 이메일 발송은 Mock 또는 로그 출력으로 대체할 수 있습니다.
- 실제 메시지 브로커는 설치하지 않되, 나중에 Kafka/RabbitMQ/SQS 등으로 교체 가능한 경계를 유지합니다.

## 검증 기준

- 요청 등록 API는 즉시 접수 응답을 반환합니다.
- 상태 조회 API는 현재 상태, 재시도 횟수, 마지막 실패 사유를 확인할 수 있습니다.
- 사용자 알림 목록 API는 수신자 기준 목록과 읽음/안읽음 필터를 지원합니다.
- 실패 시 재시도 또는 최종 실패 상태로 전이됩니다.
- 동일 이벤트 동시 요청에서 중복 데이터 또는 중복 발송이 발생하지 않습니다.
- 재시작/장시간 처리/다중 인스턴스 시나리오를 코드 또는 테스트로 설명할 수 있습니다.
